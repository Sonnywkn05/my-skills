## Context

`openspec/config.yaml` currently contains only `schema: spec-driven` with no `context` or `rules`. This means every artefact generated by OpenSpec for this repo falls back to generic defaults — there is no signal that this is a skill-authoring workspace, no enforcement of evaluation-first development, and no structural requirements for the observe-and-measure cycle described in Claude's agent skills best practices.

The evaluation-first pattern has three layers:
1. **Before writing**: a proposal must declare what will be measured (baseline)
2. **During writing**: specs must include evaluation scenarios (what "good" looks like)
3. **After writing**: tasks must verify behavior via a subagent observe-and-measure cycle before marking work complete

None of these are currently enforced.

## Goals / Non-Goals

**Goals:**
- Add a `context` block that frames this as a skill-authoring repo with evaluation-first norms and a conciseness constraint
- Add prescriptive `rules` for `proposal`, `specs`, and `tasks` that enforce the three layers above
- Make the observe-and-measure task structure explicit enough that an agent executing the tasks can follow it mechanically without interpretation

**Non-Goals:**
- Changing the OpenSpec schema or adding new artefact types
- Enforcing rules retroactively on existing or archived changes
- Defining actual evaluation content (that belongs in individual specs and tasks)
- Supporting changes that are not about skill authoring (this config is repo-scoped)

## Decisions

### Decision 1: `context` for philosophy, `rules` for structure

The `context` field is injected into every artefact generation prompt as background. The `rules` field adds artefact-specific constraints. Rather than trying to encode everything in one place, the split is:

- `context`: why this repo exists, what evaluation-first means, conciseness as a first-class constraint
- `rules.proposal`: require a Baseline Measurement section before work begins
- `rules.specs`: require ≥3 evaluation scenarios in the structured JSON format
- `rules.tasks`: require the Implement → Observe → Measure sub-step structure on any task touching skill content

**Alternative considered**: encoding all guidance in `context` only. Rejected — context is passive framing; `rules` are active constraints that the generating agent must satisfy.

### Decision 2: Prescriptive over flexible for tasks

The tasks rule will specify the exact three sub-steps (Implement, Observe, Measure) with required fields for each, rather than saying "include a verification step" and leaving interpretation to the agent.

**Rationale**: The contamination problem (an agent checking its own work) requires the Observe step to explicitly involve a fresh subagent. A vague rule like "verify behavior" would not communicate this clearly enough to be reliably followed.

**Alternative considered**: flexible framing ("tasks are not complete until behavior is verified"). Rejected — too easy for the generating agent to produce a nominal verification step that doesn't actually use a subagent.

### Decision 3: Evaluation JSON format in specs

Specs must include evaluation scenarios in the structured JSON format from the Claude docs:
```json
{
  "query": "...",
  "expected_behavior": ["...", "..."]
}
```

**Rationale**: This format is machine-readable and directly maps to the observe step in tasks. Having them in specs means the test queries and expected signals are defined before any implementation begins.

### Decision 4: Proposal baseline gates the start of work

The `rules.proposal` requirement to include a Baseline Measurement section means the author must define what Claude does *without* the skill (or the change) before writing a single line of skill content. This is the "evaluate first" principle from the best practices doc.

**Rationale**: Without a declared baseline, there's no way to know if the change actually improved anything. This is a soft gate — it's enforced by the rule during artefact generation, not by the CLI.

## Risks / Trade-offs

- **Tasks become heavier** — the three sub-steps add length to each task that touches skill content. Mitigation: the rules scope this only to tasks that write or modify skill content, not all tasks.
- **Subagent observation requires execution context** — the observe step assumes the agent executing tasks can invoke a subagent. If the execution environment doesn't support this, the observe step becomes a manual checklist. Mitigation: document the fallback ("if subagent not available, manually test in a fresh session").
- **Evaluation JSON in specs may feel mechanical** — authors may write pro-forma scenarios that don't reflect real usage. No mitigation other than code review; this is a content quality problem, not a structural one.
